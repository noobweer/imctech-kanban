# 3. Продуктовая логика комментариев

У каждой `Task` есть своя плоская ветка комментариев.

Плоская ветка означает:

```text
comment 1
comment 2
comment 3
```

Не делать вложенные ответы:

```text
comment
  reply
    reply
```

# 5. Кто может создавать comments

Правила создания комментариев зависят от того, есть ли уже комментарии у задачи.

## 5.1. Первый комментарий

Если у задачи ещё нет не удалённых комментариев, первый комментарий может создать только:

- mentor с доступом к доске;
- staff/superuser.

Student не может начать ветку сам.

Assignee не может начать ветку сам.

Owner задачи не может начать ветку сам, если он не mentor/staff.

## 5.2. Последующие комментарии

Если у задачи уже есть хотя бы один не удалённый комментарий, писать в ветку могут:

- mentor с доступом к доске;
- assignee этой задачи;
- staff/superuser.

Owner задачи не получает право писать автоматически.

Если owner задачи не находится в `task.assignees`, он не может писать комментарии.

Это важно: участие в ветке после старта даётся ответственным за задачу, а не создателю задачи.

# 6. Edit/Delete comments

Редактирование комментария:

- author может редактировать свой комментарий;
- staff/superuser может редактировать любой комментарий;
- mentor не может редактировать чужой комментарий, если он не staff;
- assignee не может редактировать чужой комментарий.

Удаление комментария:

- author может удалить свой комментарий;
- staff/superuser может удалить любой комментарий;
- mentor не может удалить ответ студента, если он не author/staff;
- assignee не может удалить комментарий наставника.

Удаление делать soft-delete.

Не удалять запись из БД физически.

# 7. Модель `TaskComment`

```python
class TaskComment(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    task = models.ForeignKey(Task, on_delete=models.CASCADE, related_name="comments")
    owner = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name="task_comments")
    content = models.TextField()
    links = models.JSONField(default=list, blank=True)
    is_deleted = models.BooleanField(default=False)
    deleted_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

Правила:

- `content` обязателен;
- пустой `content` не разрешать;
- `links` — JSON-list строк;
- если `links` не передан, сохранять `[]`;
- `is_deleted=False` по умолчанию;
- обычные API list/detail не должны возвращать soft-deleted comments как активные сообщения.

`Meta`:

```python

    indexes = [
        models.Index(fields=["task", "created_at"]),
        models.Index(fields=["owner", "created_at"]),
```

# 8. Модель `TaskCommentReadState`

Добавить модель в `pb/models.py`.

Назначение: хранить, до какого момента конкретный пользователь прочитал комментарии конкретной задачи.

```python
class TaskCommentReadState(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    task = models.ForeignKey(Task, on_delete=models.CASCADE, related_name="comment_read_states")
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name="comment_read_states")
    last_read_at = models.DateTimeField()
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

`Meta`:

```python
class Meta:
    unique_together = ("task", "user")
    indexes = [
        models.Index(fields=["task", "user"]),
        models.Index(fields=["user", "last_read_at"]),
    ]
```

Правила:

- одна запись на пару `(task, user)`;
- запись создаётся или обновляется через mark-read endpoint;
- собственные комментарии пользователя не должны считаться непрочитанными для него.

# 22. Расчёт comment state

Для task и current user считать:

```text
comments_count = count(TaskComment where task=task and is_deleted=False)
last_comment_at = max(created_at) среди not deleted comments
has_comments = comments_count > 0
```

Если `comments_count == 0`:

```json
{
  "comments_count": 0,
  "has_comments": false,
  "has_unread_comments": false,
  "last_comment_at": null,
  "comments_state": "none"
}
```

Если комментарии есть:

1. Найти `TaskCommentReadState` для `(task, user)`.
2. Если read state отсутствует:
   - если все comments написаны самим user -> `read`;
   - если есть comments от других пользователей -> `unread`.
3. Если read state есть:
   - `unread`, если существует not deleted comment:
     - `created_at > last_read_at`;
     - `owner != user`;
   - иначе `read`.

Собственные комментарии пользователя не считаются непрочитанными для него.

Edit/delete комментария не создаёт unread-state.

Unread создаётся только новым comment от другого пользователя.

# 23. Визуальное выделение на frontend

Backend должен отдавать достаточные данные:

```json
{
  "task_id": "uuid",
  "comments_count": 3,
  "has_comments": true,
  "has_unread_comments": true,
  "last_comment_at": "2026-05-31T12:00:00Z",
  "comments_state": "unread"
}
```

Frontend может использовать:

```text
comments_state = none   -> нет выделения
comments_state = read   -> нейтральное выделение / значок comments
comments_state = unread -> яркое выделение / индикатор новых comments
```

Не добавлять `comments_state` как обычное поле модели `Task`, потому что read/unread зависит от конкретного пользователя.

1. Вызвать:

```text
GET /api/boards/{board_id}/comments/states
```

Ожидаемо:

- возвращаются states только для tasks этой board;
- для задач без comments `comments_state=none`;
- для задач с прочитанными comments `comments_state=read`;
- для задач с непрочитанными comments `comments_state=unread`.

# 27. Автоматические тесты

Добавить tests, если в проекте уже есть тестовая инфраструктура.

Минимальный набор:

- mentor can create first comment;
- assignee cannot create first comment;
- assignee can reply after mentor comment;
- task owner cannot reply if not assignee;
- board member can read comments;
- board member not assignee cannot add comments;
- author can edit own comment;
- author can soft delete own comment;
- user cannot edit/delete another user’s comment;
- mark-read creates read state;
- new comment from another user makes state unread;
- own comment does not make state unread for author;
- board comment states return correct values.

Если тестовой инфраструктуры нет, описать ручную Swagger-проверку в документации.
