# 2. Что нужно реализовать

Реализовать сущность `Invite` и backend-логику для управления командой доски.

Фронтовый сценарий, который должен поддерживаться backend:

- пользователь открывает `Members`;
- owner доски или staff/admin может сгенерировать invite-ссылку;

- у invite есть max uses, включая вариант `Unlimited`;

- пользователь может перейти по invite и присоединиться к доске;
- в модалке отображается список текущих участников;

Важно: если endpoint’ы members уже есть в проекте, не создавать дубликаты. Переиспользовать и доработать существующие endpoint’ы под требования ниже. Если их нет — добавить.

# 3. Почему Board.members и members endpoint — не дублирование

`Board.members` — это поле модели и связь в базе. Оно хранит, какие пользователи состоят в доске, и используется для проверки доступа.

Endpoint’ы members — это API-слой для frontend:

- показать участников в модалке;
- показать owner;
- показать роли;
- исключить участника;
- выйти из команды.

То есть `Board.members` — хранение и права, а `/boards/{board_id}/members` — интерфейс доступа к этим данным через API.

# 4. Модель Invite

Добавь модель `Invite`.

Поля:

- `id` — UUID, primary key. Используется как invite-код.
- `board` — ForeignKey на `Board`.
- `max_uses` — nullable positive integer. Если `null`, invite безлимитный по количеству использований.
- `used_count` — integer, по умолчанию `0`.
- `expire_at` — datetime. Время истечения invite.
- `is_active` — boolean, по умолчанию `true`.
- `created_by` — ForeignKey на пользователя, который создал invite.
- `created_at` — datetime.
- `updated_at` — datetime.

Важно:

- `max_uses = null` означает `Unlimited`.
- `used_count` увеличивается только при успешном новом присоединении пользователя.
- Повторное присоединение того же пользователя НЕ увеличивает `used_count`.
- `DELETE invite` не удаляет invite физически, а деактивирует его через `is_active = false`.
- Для MVP один активный invite на доску предпочтительнее множества активных ссылок.

# 5. Поведение Generate New Link

Frontend имеет кнопку `Generate New Link`.

Реализовать так:

`POST /api/boards/{board_id}/invites`

Создаёт новую invite-ссылку для доски.

При создании нового invite:

- текущий пользователь должен быть owner доски или staff/admin;
- если у доски уже есть активные invite-ссылки, деактивировать старые (`is_active = false`);
- создать новый active invite;
- вернуть `invite_url` или `invite_code`, чтобы frontend мог показать и скопировать ссылку;
- `used_count = 0`;
- `created_by = request.user`.

Почему так:

- во frontend показано одно поле invite link;
- кнопка `Generate New Link` логически заменяет старую ссылку новой;
- старые ссылки безопаснее отключать.

# 6. Request/response для создания invite

Endpoint:

`POST /api/boards/{board_id}/invites`

Тело запроса:

```json
{
  "expires_in_days": 7,
  "max_uses": null
}
```

Правила:

- `expires_in_days` — положительное число;
- если `expires_in_days` не передан, использовать `7`;
- `expire_at = now + expires_in_days`;
- `max_uses` может быть:
  - положительным числом;
  - `null`, если выбран `Unlimited`;
- `max_uses = 0` запрещён;
- `max_uses < used_count` запрещён при обновлении.

Ответ:

```json
{
  "id": "uuid",
  "board_id": "uuid",
  "board_name": "Board name",
  "invite_code": "uuid",
  "invite_url": "https://frontend-url/join/uuid",
  "max_uses": null,
  "used_count": 0,
  "expire_at": "2026-05-17T20:00:00Z",
  "is_active": true,
  "created_by_username": "test_mentor",
  "created_at": "2026-05-17T10:00:00Z",
  "updated_at": "2026-05-17T10:00:00Z"
}
```

Если frontend base URL неизвестен, возвращай хотя бы:

```json
{
  "invite_code": "uuid",
  "invite_path": "/join/uuid"
}
```

И укажи это в отчёте.

# 8. Правила endpoint’ов Invite

## `GET /api/boards/{board_id}/invites`

Возвращает историю invite-ссылок доски.

Доступ:

- owner доски;
- staff/admin.

Обычный member не должен видеть список invite-ссылок.

Ответ: список invite с полями:

- `id`
- `invite_code`
- `invite_url` или `invite_path`
- `max_uses`
- `used_count`
- `expire_at`
- `is_active`
- `created_by_username`
- `created_at`
- `updated_at`

## `GET /api/boards/{board_id}/invites/current`

Возвращает текущий активный invite доски.

Доступ:

- owner доски;
- staff/admin.

Если активного invite нет — вернуть `404` или понятный ответ.

## `POST /api/boards/{board_id}/invites`

Создаёт новый invite и деактивирует старые active invite этой доски.

Доступ:

- owner доски;
- staff/admin.

Обычный member не может генерировать invite.

## `GET /api/invites/{invite_id}`

Возвращает invite.

Для owner/staff можно вернуть полную информацию.

Для обычного авторизованного пользователя, который хочет присоединиться, достаточно вернуть безопасную информацию:

- существует ли invite;
- активен ли invite;
- не истёк ли invite;
- название доски;
- имя owner/команды, если это уже используется во frontend.

## `PATCH /api/invites/{invite_id}`

Позволяет изменить:

- `expires_in_days` или `expire_at`;
- `max_uses`;
- `is_active`.

Доступ:

- owner доски;
- staff/admin.

Проверки:

- `max_uses` не может быть `0`;
- если `max_uses` не `null`, оно не может быть меньше `used_count`;
- `expire_at` должен быть в будущем, если `is_active = true`.

## `DELETE /api/invites/{invite_id}`

Деактивирует invite:

```text
is_active = false
```

Физически не удалять.

Доступ:

- owner доски;
- staff/admin.

## `POST /api/invites/{invite_id}/join`

Добавляет текущего пользователя в `Board.members`.

Доступ:

- любой авторизованный пользователь.

Проверки:

- invite существует;
- invite `is_active = true`;
- `expire_at` не в прошлом;
- если `max_uses != null`, то `used_count < max_uses`;
- пользователь ещё не является owner доски;
- пользователь ещё не находится в `Board.members`.

Если пользователь уже owner или member:

- не добавлять повторно;
- не увеличивать `used_count`;
- вернуть понятный ответ.

При успешном join:

- добавить пользователя в `board.members`;
- увеличить `used_count` на `1`;
- вернуть данные доски или краткий success response.

# 9. Endpoint’ы Members

## `GET /api/boards/{board_id}/members`

Возвращает список участников доски.

Доступ:

- owner;
- member;
- staff/admin.

Ответ должен подходить для members modal:

```json
[
  {
    "username": "test_mentor",
    "name": "Test Mentor",
    "role": "mentor",
    "is_owner": true
  },
  {
    "username": "test_student",
    "name": "Test Student",
    "role": "student",
    "is_owner": false
  }
]
```

Важно:

- owner должен быть в ответе;
- members должны быть в ответе;
- не возвращать password/hash.

## `DELETE /api/boards/{board_id}/members/{username}`

Исключает участника из доски.

Доступ:

- owner доски;
- staff/admin.

Проверки:

- нельзя исключить owner через этот endpoint;
- нельзя исключить пользователя, которого нет в members;
- нельзя исключать участников чужой доски без прав;
- после исключения пользователь больше не должен видеть доску.

## `POST /api/boards/{board_id}/leave`

Позволяет текущему пользователю выйти из команды.

Доступ:

- авторизованный пользователь, который состоит в members доски.

Проверки:

- owner не может выйти через этот endpoint;
- если owner хочет выйти, нужна передача ownership, но её в этой задаче НЕ реализовывать;
- если пользователь не member доски — вернуть понятную ошибку;
- после выхода пользователь удаляется из `board.members`.

# 10. Права доступа

Использовать простую модель:

- staff/admin может управлять всеми invite и members;
- owner доски может:
  - генерировать invite;
  - смотреть invite;
  - обновлять invite;
  - отзывать invite;
  - смотреть участников;
  - исключать участников;
- member доски может:
  - смотреть список участников;
  - выйти из команды;
- member доски НЕ может:
  - генерировать invite;
  - обновлять invite;
  - отзывать invite;
  - исключать других;
- неавторизованный пользователь не может использовать API.

Роли `student` и `mentor` пока использовать только как отображаемые роли из UserProfile.  
Не строить сложную permission-систему на student/mentor.

---

# 11. Валидация

Добавить проверки:

- `expires_in_days` — положительное число;
- `expire_at` не должен быть в прошлом при active invite;
- `max_uses`:
  - `null` допустимо и означает Unlimited;
  - число должно быть положительным;
  - не может быть меньше `used_count`;
- `used_count >= 0`;
- если `max_uses != null`, то `used_count <= max_uses`;
- inactive invite нельзя использовать;
- expired invite нельзя использовать;
- invite с исчерпанным лимитом нельзя использовать;
- нельзя добавить одного пользователя в members дважды;
- повторный join не увеличивает `used_count`;
- нельзя удалить owner из доски через remove-member;
- owner не может выйти через leave;
- нельзя управлять invite чужой доски без прав owner/staff.

# 15. Ручная проверка через Swagger

## Предусловия

Есть owner:

```json
{
  "username": "test_mentor",
  "password": "12345678",
  "name": "Test Mentor",
  "role": "mentor"
}
```

Есть student:

```json
{
  "username": "test_student",
  "password": "12345678",
  "name": "Test Student",
  "role": "student"
}
```

Есть доска:

```json
{
  "name": "Test Board"
}
```

Все запросы выполнять с JWT:

```text
Authorization: Bearer <access_token>
```

## Проверить invite generation

1. Под owner JWT вызвать:

```text
POST /api/boards/{board_id}/invites
```

Тело:

```json
{
  "expires_in_days": 7,
  "max_uses": null
}
```

2. Проверить, что ответ содержит:

- `invite_code`;
- `invite_url` или `invite_path`;
- `max_uses = null`;
- `used_count = 0`;
- `is_active = true`;
- `expire_at`.

3. Повторно создать invite и проверить:

- старый active invite стал inactive;
- новый invite active.

## Проверить join

1. Под student JWT вызвать:

```text
POST /api/invites/{invite_id}/join
```

2. Проверить:

- student добавлен в `Board.members`;
- `used_count` увеличился на `1`.

3. Повторить join тем же student.

Ожидаемо:

- дубликата в members нет;
- `used_count` не увеличился.

## Проверить members modal

Вызвать:

```text
GET /api/boards/{board_id}/members
```

Проверить:

- owner есть в списке;
- student есть в списке;
- у owner `is_owner = true`;
- роли `student/mentor` возвращаются;
- password/hash не возвращается.

## Проверить remove member

Под owner JWT:

```text
DELETE /api/boards/{board_id}/members/test_student
```

Проверить:

- student удалён из members;
- student больше не видит доску;
- owner не может быть удалён этим endpoint’ом.

## Проверить leave

1. Снова добавить student через invite.
2. Под student JWT вызвать:

```text
POST /api/boards/{board_id}/leave
```

Проверить:

- student удалён из members;
- student больше не видит доску.

Под owner JWT вызвать leave.

Ожидаемо:

- ошибка;
- owner не выходит из доски.
