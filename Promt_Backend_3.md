# Сущность Column

Реализуй `Column` как сущность колонки Kanban-доски.

Поля:

- `id` — UUID, primary key;
- `board` — ForeignKey на `Board`;
- `name` — название колонки;
- `position` — порядок отображения колонки внутри доски;
- `status` — статус колонки:
  - `active`
  - `archived`
- `sum_tasks` — количество задач в колонке, по умолчанию `0`;
- `created_at`;
- `updated_at`.

Важно:

- `Column` должна относиться к конкретной доске.
- `position` нужен для правильного порядка отображения колонок.
- `sum_tasks` пока является числовым полем-заглушкой.
- Не реализовывать автоматический подсчёт задач, потому что сущность `Task` не входит в эту задачу.
- Не реализовывать frontend drag-and-drop.

# Дефолтные колонки

Реализовать отдельный endpoint:

```text
POST /api/boards/{board_id}/columns/defaults
```

Он создаёт дефолтные колонки:

- `To Do`, position `1`;
- `In Progress`, position `2`;
- `Done`, position `3`.

Правила:

- endpoint должен работать только для доски, к которой у пользователя есть доступ;
- создавать дефолтные колонки может только owner доски или staff/admin;
- если у доски уже есть хотя бы одна колонка — не создавать дубликаты, а вернуть понятную ошибку;
- не менять логику создания доски из задачи 2;
- не добавлять автоматическое создание колонок в `POST /api/boards`, если этот endpoint уже реализован и работает.

В отчёте указать, что выбран безопасный вариант через отдельный endpoint.

# Правила работы endpoint’ов

## Получение списка колонок

`GET /api/boards/{board_id}/columns`

Возвращает колонки конкретной доски.

Требования:

- возвращать только колонки указанной доски;
- сортировать по `position ASC`;
- пользователь должен иметь доступ к доске;
- желательно поддержать фильтр:
  - `GET /api/boards/{board_id}/columns?status=active`
  - `GET /api/boards/{board_id}/columns?status=archived`

---

## Получение одной колонки

`GET /api/columns/{column_id}`

Возвращает одну колонку.

Требования:

- пользователь должен иметь доступ к доске, к которой относится колонка.

Ответ по колонке должен включать:

- `id`
- `board_id`
- `board_name`
- `name`
- `position`
- `status`
- `sum_tasks`
- `created_at`
- `updated_at`

---

## Создание колонки

`POST /api/boards/{board_id}/columns`

Создаёт колонку внутри доски.

Минимальное тело запроса:

```json
{
  "name": "Review"
}
```

Если `position` не передан, колонка добавляется в конец списка.

Можно поддержать явный `position`:

```json
{
  "name": "Review",
  "position": 2
}
```

Если реализуется явный `position`, нужно корректно пересчитать позиции остальных колонок этой же доски.

Создавать колонку может только owner доски или staff/admin.

---

## Обновление колонки

`PATCH /api/columns/{column_id}`

Обновляет колонку.

Можно менять:

- `name`;
- `status`.

Если перемещение через `PATCH` не реализуется, `position` менять только через:

```text
POST /api/columns/{column_id}/move
```

Пример:

```json
{
  "name": "Code Review",
  "status": "active"
}
```

Обновлять колонку может только owner доски или staff/admin.

---

## Перемещение колонки

`POST /api/columns/{column_id}/move`

Меняет порядок колонки.

Тело запроса:

```json
{
  "position": 1
}
```

Логика:

- изменить позицию выбранной колонки;
- пересчитать позиции остальных колонок этой же доски;
- не затрагивать колонки других досок;
- не оставлять дублирующиеся позиции;
- позиции после перемещения должны быть последовательными.

Пример:

Было:

```text
1. To Do
2. In Progress
3. Done
```

`Done` перемещают на позицию `1`.

Должно стать:

```text
1. Done
2. To Do
3. In Progress
```

Перемещать колонку может только owner доски или staff/admin.

---

## Архивация колонки

`POST /api/columns/{column_id}/archive`

Архивирует колонку.

Логика:

- поставить `status = archived`;
- колонка остаётся в базе;
- активный список колонок может её не показывать при фильтре `status=active`.

Архивировать колонку может только owner доски или staff/admin.

---

## Удаление колонки

`DELETE /api/columns/{column_id}` не должен физически удалять колонку.

В этой задаче удаление реализовать как безопасную архивацию:

- поставить `status = archived`;
- оставить колонку в базе;
- не удалять связанные данные.

Физическое удаление не делать, потому что в будущих задачах у колонки будут связанные задачи.

В отчёте указать, что `DELETE` реализован как soft-delete через архивирование.

---

# Логика position

Позиции колонок внутри одной доски должны быть последовательными:

```text
1, 2, 3, 4...
```

Правила:

- `position` всегда положительное число;
- при создании без `position` колонка добавляется в конец;
- при перемещении колонок пересчитывать позиции только внутри этой же доски;
- не затрагивать колонки других досок;
- после перемещения не должно быть дублей `position`;
- активные колонки одной доски не должны иметь одинаковый `position`.

Если проще технически — можно поддерживать уникальность `(board, position)` для всех колонок, включая archived.

Если выбрана другая стратегия — описать её в отчёте.

---

# Валидация

Добавить проверки:

- название колонки не должно быть пустым;
- `status` должен быть только `active` или `archived`;
- `position` должен быть положительным числом;
- `sum_tasks >= 0`;
- нельзя получить колонки доски без доступа к этой доске;
- нельзя менять колонки доски без прав owner/staff;
- нельзя перемещать колонку за пределы своей доски;
- после перемещения позиции колонок внутри доски должны быть последовательными;
- не должно быть двух активных колонок одной доски с одинаковой `position`.

---

# Права доступа

Сделать простую логику:

- все endpoint’ы должны требовать JWT;
- читать колонки доски может пользователь, у которого есть доступ к доске;
- пользователь имеет доступ к доске, если он `owner`;
- пользователь имеет доступ к доске, если он есть в `members`;
- staff/admin имеет доступ ко всем доскам;
- создавать, обновлять, перемещать, архивировать и удалять колонки может owner доски или staff/admin;
- обычный участник доски может читать колонки, но не может менять структуру доски.

Роли `student` и `mentor` пока не усложнять. Они нужны для будущей логики.
