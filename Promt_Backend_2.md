- Не усложнять архитектуру.
- Не реализовывать `Column`, `Task`, `Comment`, `Invite`, `Activity` в этой задаче.

# Сущность Board

Реализуй `Board` как Kanban-доску, которая будет отображаться на дашборде пользователя.

Поля:

- `id` — UUID, primary key;
- `name` — название доски;
- `project` — ForeignKey на `Project`;
- `owner` — ForeignKey на пользователя;
- `members` — ManyToMany к пользователям;
- `status` — статус доски:
  - `active`
  - `archived`
- `tasks_total` — общее количество задач, по умолчанию `0`;
- `tasks_done` — количество выполненных задач, по умолчанию `0`;
- `progress_percent` — процент выполнения, по умолчанию `0`;
- `created_at`;
- `updated_at`.

Важно:

- `tasks_total`, `tasks_done`, `progress_percent` пока являются числовыми полями-заглушками.
- Не реализовывать автоматический подсчёт задач, потому что сущность `Task` не входит в эту задачу.
- Не реализовывать `Column` в этой задаче.
- Не создавать дефолтные колонки при создании доски. Это относится к задаче 3.
- Архитектуру `Board` сделать так, чтобы позже можно было безопасно связать её с `Column`.

---

# Валидация Board

Для числовых полей:

- `tasks_total >= 0`;
- `tasks_done >= 0`;
- `tasks_done <= tasks_total`, если оба значения передаются вручную;
- `progress_percent` должен быть в диапазоне `0..100`.

По умолчанию все три поля равны `0`.

# Правила работы endpoint’ов

## Project

`GET /api/projects`

- возвращает список проектов, доступных текущему пользователю;
- staff/admin видит все проекты.

`GET /api/projects/{project_id}`

- возвращает конкретный проект, если у пользователя есть к нему доступ;
- пользователь имеет доступ к проекту, если у него есть доступ хотя бы к одной доске этого проекта;
- staff/admin имеет доступ ко всем проектам.

`POST /api/projects`

- создаёт проект;
- доступен авторизованному пользователю.

`PATCH /api/projects/{project_id}`

- обновляет проект;
- пока достаточно обновления `name`;
- доступен staff/admin или пользователю, который является owner хотя бы одной доски проекта.

`DELETE /api/projects/{project_id}`

- должен быть безопасным;
- если у проекта нет досок — можно удалить проект;
- если у проекта есть связанные доски — не удалять физически, а вернуть понятную ошибку;
- не делать каскадное удаление досок без явного безопасного решения;
- в отчёте указать выбранную стратегию удаления.

Ответ по проекту должен включать:

- `id`
- `name`
- `created_at`
- `updated_at`

---

## Board

`GET /api/boards`

- возвращает список досок, доступных текущему пользователю;
- пользователь видит доску, если он `owner`;
- пользователь видит доску, если он есть в `members`;
- staff/admin видит все доски.

`GET /api/boards/{board_id}`

- возвращает конкретную доску, если у пользователя есть доступ.

`POST /api/boards`

- создаёт доску;
- текущий авторизованный пользователь становится `owner`;
- текущий авторизованный пользователь автоматически добавляется в `members`;
- статус доски по умолчанию `active`;
- если `project_id` передан — доска создаётся внутри существующего проекта;
- если `project_id` не передан — можно автоматически создать проект с таким же или похожим названием;
- не создавать дефолтные колонки в этой задаче.

`PATCH /api/boards/{board_id}`

- обновляет доску;
- менять можно только:
  - `name`;
  - `status`;
- редактировать может только `owner` или staff/admin.

`DELETE /api/boards/{board_id}`

- удаляет доску или архивирует её, если выбран более безопасный вариант;
- предпочтительно сделать безопасную стратегию: `status = archived`;
- в отчёте указать выбранную стратегию.

Ответ по доске должен включать:

- `id`
- `name`
- `project_id`
- `project_name`
- `owner_username`
- `members`
- `status`
- `tasks_total`
- `tasks_done`
- `progress_percent`
- `created_at`
- `updated_at`

В `members` можно возвращать список username.

# Права доступа

Сделать простую логику:

- все endpoint’ы должны требовать JWT;
- пользователь видит доску, если он `owner`;
- пользователь видит доску, если он есть в `members`;
- staff/admin видит все доски;
- пользователь видит проект, если у него есть доступ хотя бы к одной доске этого проекта;
- staff/admin видит все проекты;
- создавать проект может авторизованный пользователь;
- создавать доску может авторизованный пользователь;
- обновлять/удалять доску может `owner` или staff/admin;
- обычный member может читать доску, но не может её редактировать.

Роли `student` и `mentor` пока не усложнять. Они нужны для будущей логики.
