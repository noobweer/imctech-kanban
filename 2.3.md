# 3. Добавить ColumnKind.ARCHIVE

Расширить `ColumnKind`.

Должно быть:

```python
class ColumnKind(models.TextChoices):
    BOARD = "board", "Board"
    BACKLOG = "backlog", "Backlog"
    ARCHIVE = "archive", "Archive"
```

Техническая Archive column:

```text
name = "Archive"
kind = "archive"
status = "active"
```

Правила:

- у каждой board максимум одна active Archive column;
- Archive column не показывается в обычном board view;
- Archive column не показывается в backlog view;
- Archive column используется только для хранения архивных задач;
- пользователь не должен создавать Archive column через обычный create column endpoint;
- backend создаёт Archive column автоматически, когда нужно архивировать первую задачу на доске.

# 4. Общая логика архива задач

Task archive/delete = перенос задачи в техническую Archive column.

Не делать:

```text
Task.status = archived
```

Делать:

```text
task.column = archive_column
task.position = конец archive_column
```

Сама Task остаётся в базе.

Физически задачи не удалять.

# 5. Restore task

Пользователь сам выбирает, куда восстановить задачу.

Endpoint:

```text
POST /api/tasks/{task_id}/restore
```

Request:

```json
{
  "target_column_id": "uuid",
  "position": 1
}
```

`position` опционален. Если не передан — восстановить в конец выбранной колонки.

Backend не должен угадывать старую колонку и не должен требовать `archived_from_column`.

Проверки:

- task должна сейчас находиться в Archive column;
- target column должна существовать;
- target column должна принадлежать той же board;
- target column не должна быть archived;
- target column.kind должен быть `board` или `backlog`;
- target column.kind не может быть `archive`;
- пользователь должен иметь доступ к board.

После restore:

- перенести task в target column;
- выставить position;
- пересчитать positions в Archive column;
- пересчитать positions в target column;
- обновить `sum_tasks` у затронутых колонок;
- пересчитать progress в базе, если progress хранится в Board;
- вернуть обновлённую task и affected columns/ids.

# 7. Получение или создание Archive column

Добавить helper/service функцию:

```text
get_or_create_archive_column(board)
```

Логика:

1. Найти active column:
   - `board = board`;
   - `kind = archive`;
   - `status = active`.
2. Если найдена — использовать её.
3. Если не найдена — создать:
   - `name = "Archive"`;

4. Не создавать вторую active Archive column для одной board.

Archive column position не должна конфликтовать с positions обычных board/backlog columns, потому что это отдельный `kind`.

# 11. Column archive

Колонки архивируются не через Archive column.

Архив колонки:

```text
Column.status = archived
```

Доработать существующий endpoint:

```text
POST /api/columns/{column_id}/archive
```

Логика:

1. Проверить JWT.
2. Получить column.
3. Проверить права.
4. Если column.kind = backlog, не архивировать.
5. Если column.kind = archive, не архивировать.
6. Если column уже archived, вернуть текущий ColumnOut.
7. Если column active:
   - поставить `status = archived`;
   - не менять task.column у задач внутри;
   - пересчитать positions active columns того же board/kind;
   - пересчитать progress в базе, если progress хранится в Board.
8. Вернуть ColumnOut.

Важно:

- archive column НЕ переносит задачи в Archive column;
- задачи остаются внутри этой column;
- они скрываются из обычной доски, потому что parent column archived;
- при restore column они снова станут видимыми.

# 14. Clear column

Добавить endpoint:

```text
POST /api/columns/{column_id}/clear
```

Request body можно оставить пустым.

Назначение:

- очистить колонку от задач;
- все задачи из этой column перенести в Archive column;
- саму column оставить active.

Логика:

1. Проверить JWT.
2. Получить column.
3. Проверить доступ к board column.
4. Проверить, что column.status = active.
5. Проверить, что column.kind = board или backlog.
6. Получить или создать Archive column.
7. Внутри `transaction.atomic()`:
   - перенести все задачи из column в Archive column;
   - поставить перенесённые задачи в конец Archive column;
   - пересчитать positions source column;
   - пересчитать positions Archive column;
   - поставить `source_column.sum_tasks = 0`;
   - обновить `archive_column.sum_tasks`;
   - пересчитать progress в базе, если progress хранится в Board.
8. Вернуть response:

```json
{
  "success": true,
  "archived_tasks_count": 5,
  "affected_column_ids": [
    "source-column-id",
    "archive-column-id"
  ]
}
```

Clear column должен быть атомарным: либо все задачи перенесены, либо ничего не изменено.

В конце дай отчёт:

1. Какие файлы изменены.
2. Какие миграции созданы.
3. Как добавлен `ColumnKind.ARCHIVE`.
4. Как создаётся technical Archive column.
5. Как работает archive task.
6. Как работает restore task.
7. Как работает DELETE task.
8. Как работает archive column.
9. Как работает restore column.
10. Как работает DELETE column.
11. Как работает clear column.
12. Как работают archive views.
13. Как фильтруются active lists.
14. Как пересчитывается `Column.sum_tasks`.
15. Как пересчитывается board progress в базе.
16. Как устроены права members.
17. Как проверить всё через Swagger.
18. Какие ограничения остались.
