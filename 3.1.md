Ты работаешь в существующей backend-кодовой базе Kanban-модуля IMCTech.

Нужно реализовать Sprint 3 Task 1: разделить права доступа ролей `mentor` и `student`.

Задача: 
- Наставник (`mentor`) выступает в роли организатора (создает проекты/доски, колонки, приглашает участников), но не должен иметь доступа к созданию, редактированию, перемещению или архивированию самих карточек задач. Наставник может только просматривать задачи. Любой приглашенный на доску ментор становится её совладельцем (co-manager).
- Студент (`student`) работает с задачами, но не должен иметь прав на создание проектов и досок (соответственно, не может ими управлять).

# 2. Продуктовое правило

`mentor` и `student` — это роли пользователя.

Правило для Sprint 3 Task 1:

```text
mentor организует пространство (создает проекты, доски, колонки, инвайты);
mentor, приглашенный на чужую доску, получает права управления доской наравне с владельцем;
mentor может читать все данные доски (включая задачи);
mentor не может создавать, редактировать, перемещать, архивировать, назначать задачи и управлять чек-листами.

student работает с задачами на доске;
student может чистить колонки (clear column);
student не может создавать проекты и доски (соответственно, он не может стать владельцем и не может изменять настройки доски или инвайты).
```

Если пользователь `is_staff` или `is_superuser`, staff/admin права сильнее любых ограничений ролей.

```python
def get_user_role(user) -> str | None:
    profile = getattr(user, "profile", None)
    return getattr(profile, "role", None)

def is_mentor(user) -> bool:
    if is_staff_or_superuser(user):
        return False
    return get_user_role(user) == "mentor"

def is_student(user) -> bool:
    if is_staff_or_superuser(user):
        return False
    return get_user_role(user) == "student"
```

# 4. Permissions для структуры (Mentor = Co-manager)

Текущие функции `can_edit_board`, `can_edit_column` и `can_edit_project` в `pb/permissions.py` проверяют только то, является ли пользователь владельцем (owner).
Нужно расширить их, чтобы любой приглашенный участник с ролью `mentor` также получал права управления:

```python
def can_edit_board(user, board):
    if is_staff_or_superuser(user):
        return True
    if board.owner == user:
        return True
    # Любой приглашенный ментор получает права управления (co-manager)
    return is_mentor(user) and board.members.filter(id=user.id).exists()

def can_edit_column(user, column):
    # Логика делегируется на уровень доски
    return can_edit_board(user, column.board)

def can_edit_project(user, project):
    if is_staff_or_superuser(user):
        return True
    if project.boards.filter(owner=user).exists():
        return True
    # Ментор может управлять проектом, если он состоит хотя бы в одной доске этого проекта
    return is_mentor(user) and project.boards.filter(members=user).exists()
```

# 6. Единые ошибки для ролей

Для действий, запрещённых из-за ролей, возвращать специфичные ошибки:

**Для Mentor (задачи):**
```json
{
  "detail": "Mentor is not allowed to modify this resource",
  "code": "MENTOR_ACTION_FORBIDDEN"
}
```

**Для Student (создание структуры):**
```json
{
  "detail": "Student is not allowed to create projects or boards",
  "code": "STUDENT_ACTION_FORBIDDEN"
}
```

HTTP status во всех случаях: `403 Forbidden`

**Реализация проверки в Routers:**
Поскольку функции вроде `can_modify_task` возвращают `bool`, делайте проверку ролей внутри блока отказа. 
Для endpoints с permission helper:
```python
if not can_modify_task(request.auth, task):
    if is_mentor(request.auth):
        return router.api.create_response(request, {"detail": "Mentor is not allowed to modify this resource", "code": "MENTOR_ACTION_FORBIDDEN"}, status=403)
    raise HttpError(403, "No permission to edit this task") # старая ошибка
```
Для endpoints без permission helper (например, `POST /projects`), проверяйте роль в начале функции:
```python
if is_student(request.auth):
    return router.api.create_response(request, {"detail": "Student is not allowed to create projects or boards", "code": "STUDENT_ACTION_FORBIDDEN"}, status=403)
```

# 9. Ручная проверка через Swagger

Подготовить тестовых пользователей (два mentor, один student). 

## 9.1. Co-manager test
Mentor A создает доску и приглашает Mentor B.
Mentor B пытается изменить название доски, создать колонку и пригласить Student C.
Ожидаемо: `200 OK` / `201 Created` (у Mentor B полные права на структуру).

## 9.2. Student restrictions
Student C пытается выполнить `POST /api/projects` или `POST /api/boards`.
Ожидаемо: `403 Forbidden` с кодом `STUDENT_ACTION_FORBIDDEN`.

## 9.3. Mentor tasks mutation
Mentor A или Mentor B пытается создать задачу, отредактировать, переместить или переключить checklist.
Ожидаемо: `403 Forbidden` с кодом `MENTOR_ACTION_FORBIDDEN`.

## 9.4. Рабочий процесс задач
Student C заходит на доску. Успешно создает задачи, перемещает их и отмечает чек-листы, делает clear column.
