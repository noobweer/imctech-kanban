# Promt Backend Sprint 4 Task 3 — Mentor Requests

## Контекст задачи

Ты работаешь с backend проекта Kanban на Django + Django Ninja.

Нужно реализовать backend-часть задачи 3 Sprint/Этап 4: запросы к наставнику по задаче.

Запрос к наставнику — это не отдельный чат и не сообщение конкретному наставнику. Это активный маркер на задаче: студенту нужна помощь или проверка.

Обсуждение происходит в уже существующих комментариях задачи (`TaskComment`). Mentor request хранит состояние обращения: кто создал запрос, по какой задаче, какой статус, кто начал разбор, каким комментарием началось обсуждение и чем запрос был закрыт.

## Основная логика

```text
student-assignee создаёт запрос к задаче
запрос получает status=open

все наставники с доступом к доске видят запрос

любой наставник с доступом к доске может ответить комментарием
этот комментарий начинает обсуждение
запрос получает status=in_progress

дальше обсуждение идёт в TaskComment внутри задачи

когда студент получил нужный ответ, он сам закрывает запрос как resolved

если запрос больше не нужен или некорректен, студент или наставник закрывает его как cancelled
```

## Главное правило активного запроса

По одной задаче может быть только один активный mentor request.

Активные статусы:

```text
open
in_progress
```

Если по задаче уже есть active request, новый создать нельзя.

Это не зависит от:

```text
кто создаёт запрос
request_type
количества assignees
```

Ошибка для frontend:

```text
ACTIVE_MENTOR_REQUEST_ALREADY_EXISTS
```

После `resolved` или `cancelled` по этой же задаче можно создать новый запрос.

## Сущность TaskMentorRequest

Добавить модель `TaskMentorRequest`.

### Поля и типы данных

| Поле | Тип данных | Nullable | Смысл |
| --- | --- | --- | --- |
| `id` | `UUIDField`, primary key | нет | Уникальный идентификатор запроса |
| `task` | `ForeignKey(Task)` | нет | Задача, к которой относится запрос |
| `created_by` | `ForeignKey(User)` | нет | Пользователь, создавший запрос |
| `request_type` | `CharField` / enum | нет | Тип запроса: помощь или проверка |
| `status` | `CharField` / enum | нет | Текущий статус запроса |
| `message` | `TextField` | нет | Текст запроса от студента |
| `started_by` | `ForeignKey(User)` | да | Наставник, который первым ответил |
| `started_comment` | `ForeignKey(TaskComment)` | да | Комментарий, с которого началось обсуждение |
| `started_at` | `DateTimeField` | да | Время первого ответа наставника |
| `closed_by` | `ForeignKey(User)` | да | Пользователь, который закрыл запрос |
| `closed_at` | `DateTimeField` | да | Время закрытия запроса |
| `close_reason` | `TextField` или `CharField` | да | Причина отмены/закрытия без решения |
| `created_at` | `DateTimeField(auto_now_add=True)` | нет | Время создания |
| `updated_at` | `DateTimeField(auto_now=True)` | нет | Время последнего обновления |

### Request type

Хранить как строковый enum.

```text
help
review
```

Смысл:

```text
help — нужна помощь
review — нужна проверка
```

### Status

Хранить как строковый enum.

```text
open
in_progress
resolved
cancelled
```

Смысл:

```text
open — запрос создан, наставник ещё не ответил
in_progress — наставник начал обсуждение комментарием
resolved — студент получил ответ и отметил запрос как решённый
cancelled — запрос отменён без решения
```

## Что не требуется в модели

В текущей логике не нужны поля:

```text
mentor
links
resolved_by
cancelled_by
resolved_at
cancelled_at
```

Причины:

```text
mentor — запрос видят все наставники с доступом к доске, он не адресуется одному человеку
links — ссылки можно вставить в message
resolved_by/cancelled_by — заменяются универсальным closed_by
resolved_at/cancelled_at — заменяются универсальным closed_at
```

## Связь с комментариями

Обсуждение запроса происходит в комментариях задачи.

Когда наставник отвечает на запрос:

```text
создаётся TaskComment
request.started_by = текущий наставник
request.started_comment = созданный комментарий
request.started_at = now
request.status = in_progress
```

`started_comment` нужен как якорь, чтобы frontend мог показать, с какого комментария начался разбор запроса.

Отдельную сущность ветки комментариев в рамках этой задачи делать не нужно. История общения остаётся в общем списке комментариев задачи.

## Права

### Create

Создать request может:

```text
student, назначенный на задачу
staff/admin
```

Проверки:

```text
пользователь имеет доступ к доске задачи
пользователь является student
пользователь находится в task.assignees
по задаче нет активного request
message не пустой
request_type валидный
```

### Read

Читать request могут:

```text
все пользователи с доступом к доске
staff/admin
```

### Respond

Ответить на request может:

```text
mentor с доступом к доске
staff/admin
```

Действие `respond` создаёт комментарий в задаче и переводит request в `in_progress`.

### Resolve

Отметить request как решённый может:

```text
created_by
staff/admin
```

Логика: только пользователь, который просил помощь, понимает, получил ли он нужный ответ.

При resolve:

```text
status = resolved
closed_by = current user
closed_at = now
```

### Cancel

Отменить request может:

```text
created_by
mentor с доступом к доске
staff/admin
```

При cancel:

```text
status = cancelled
closed_by = current user
closed_at = now
close_reason = reason из payload, если передан
```

Если request отменяет наставник, желательно требовать `close_reason`, чтобы студент понимал причину закрытия.

## REST endpoint’ы

Реализовать минимальный REST API.

```text
POST   /api/tasks/{task_id}/mentor-requests
GET    /api/tasks/{task_id}/mentor-request

GET    /api/boards/{board_id}/mentor-requests
GET    /api/mentor-requests/{request_id}

POST   /api/mentor-requests/{request_id}/respond
POST   /api/mentor-requests/{request_id}/resolve
POST   /api/mentor-requests/{request_id}/cancel
```

### POST /api/tasks/{task_id}/mentor-requests

Создаёт request по задаче.

Payload:

| Поле | Тип данных | Обязательное | Смысл |
| --- | --- | --- | --- |
| `request_type` | `string` / enum | да | `help` или `review` |
| `message` | `string` | да | Текст запроса |

Response должен вернуть созданный request.

### GET /api/tasks/{task_id}/mentor-request

Возвращает активный request по задаче.

Так как по задаче может быть только один активный request, endpoint singular.

Если активного запроса нет, вернуть понятный пустой ответ или 404 — выбрать вариант, который лучше подходит текущему стилю API проекта.

### GET /api/boards/{board_id}/mentor-requests

Возвращает список запросов по доске.

Нужен для вкладки/панели запросов.

Фильтры:

| Параметр | Тип данных | Значения |
| --- | --- | --- |
| `status` | `string` / enum | `open`, `in_progress`, `resolved`, `cancelled` |
| `request_type` | `string` / enum | `help`, `review` |
| `mine` | `boolean` | `true`, `false` |

По умолчанию желательно отдавать активные requests:

```text
open
in_progress
```

### GET /api/mentor-requests/{request_id}

Возвращает один request по id.

### POST /api/mentor-requests/{request_id}/respond

Наставник отвечает на запрос.

Payload:

| Поле | Тип данных | Обязательное | Смысл |
| --- | --- | --- | --- |
| `content` | `string` | да | Текст комментария наставника |

Действие:

```text
создать TaskComment
связать request со started_comment, если он ещё не был начат
перевести request в in_progress
вернуть обновлённый request и созданный comment
```

`respond` не закрывает request как resolved. Закрывает студент отдельным endpoint’ом `resolve`.

### POST /api/mentor-requests/{request_id}/resolve

Автор запроса закрывает request как решённый.

Payload можно не требовать.

Действие:

```text
status = resolved
closed_by = current user
closed_at = now
```

### POST /api/mentor-requests/{request_id}/cancel

Автор или наставник отменяет request.

Payload:

| Поле | Тип данных | Обязательное | Смысл |
| --- | --- | --- | --- |
| `close_reason` | `string` | нет/условно | Причина отмены |

Для отмены наставником `close_reason` желательно сделать обязательным.

Действие:

```text
status = cancelled
closed_by = current user
closed_at = now
close_reason = payload.close_reason
```
