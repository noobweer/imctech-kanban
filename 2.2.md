# 3. Принятая логика совместного редактирования

В проекте используется простая стратегия:

```text
кто последним отправил изменение — тот и прав
```

Не добавлять:

- `Task.version`;
- `Column.tasks_version`;
- `Column.version`;
- optimistic locking;
- `409 Conflict`;
- поля `task_version`, `source_column_tasks_version`, `target_column_tasks_version`.

При этом backend не должен перезаписывать лишние данные.

Главное правило:

```text
PATCH обновляет только те поля, которые пришли в request.
Остальные поля Task не трогать.
```

Пример:

```json
{
  "priority": 3
}
```

Такой request меняет только `priority`.

# 6. Checklist item structure

Checklist остаётся JSON-list внутри `Task`.

Каждый checklist item должен иметь структуру:

```json
{
  "id": "uuid-string",
  "title": "Checklist item title",
  "is_done": false,
  "position": 1
}
```

Правила:

- `id` нужен, чтобы обновлять конкретный пункт, а не элемент по индексу;
- `position` нужен для порядка пунктов;
- `title` — непустая строка;
- `is_done` — boolean;
- после удаления или reorder позиции должны быть нормализованы в `1..N`;
- `checklist_done_count` и `checklist_total_count` считать из JSON при ответе.

# 7. Нормализация существующего checklist

Если в базе уже есть checklist items старого формата:

```json
{
  "title": "Create model",
  "is_done": false
}
```

нужно привести их к новому формату.

Нормализация:

- если у item нет `id`, сгенерировать UUID string;
- если у item нет `position`, выставить позицию по текущему порядку в массиве, начиная с `1`;
- если у item нет `is_done`, выставить `false`;
- `title` должен остаться строкой;
- если item полностью невалидный, привести к безопасному формату там, где возможно, и указать спорные случаи в отчёте.

Нормализацию можно сделать миграцией или service/helper-функцией, которая приводится при чтении/изменении задачи. Выбери вариант, который лучше вписывается в текущую структуру проекта, и укажи в отчёте.

# 12. MoveTask validation

Проверки:

- endpoint требует JWT;
- пользователь должен иметь доступ к доске задачи;
- task существует;
- target column существует;
- source column = текущая `task.column`;
- source column и target column принадлежат одной доске;
- source column не archived;
- target column не archived;
- `position >= 1`;
- если `position` больше длины target list + 1, вставить в конец;
- нельзя перемещать задачу в колонку `kind=archive`;
- если task уже находится в нужной колонке на нужной позиции, вернуть текущее состояние без лишних изменений.

# 13. MoveTask implementation

MoveTask выполнять внутри:

```python
transaction.atomic()
```

Алгоритм:

1. Получить task.
2. Проверить доступ.
3. Получить source column.
4. Получить target column.
5. Получить active tasks source column.
6. Получить active tasks target column.
7. Убрать task из source list.
8. Вставить task в target list на нужную позицию.
9. Пересчитать `position` в затронутых колонках:
   - позиции должны быть `1..N`;
   - не должно быть дублей;
   - не должно быть дыр.
10. Обновить перемещаемую task:
   - `column = target_column`;
   - `position = new_position`.
11. Обновить остальные affected tasks через `bulk_update`.
12. Обновить `sum_tasks` у затронутых колонок.
13. Если в `Board` хранятся `tasks_total`, `tasks_done`, `progress_percent`, пересчитать их в базе.
14. Вернуть актуальный результат для обновления kanban-доски.

Не использовать raw SQL.

# 14. MoveTask response

После успешного move вернуть данные, нужные kanban-доске для обновления колонок.

Минимальный response:

```json
{
  "task": {
    "id": "task-id",
    "column_id": "target-column-id",
    "position": 2
  },
  "affected_column_ids": [
    "source-column-id",
    "target-column-id"
  ]
}
```

Расширенный response, если frontend хочет сразу получить обновлённые задачи без дополнительных GET-запросов:

```json
{
  "task": {},
  "affected_columns": [
    {
      "column": {},
      "tasks": []
    }
  ]
}
```

`board_progress` в move response не возвращать.

Причина: progress отображается в отдельном окне/разделе, а не в drag-and-drop интерфейсе. Kanban-экрану после move нужны обновлённые задачи и колонки, а не статистика progress.

# 17. Board progress

После move backend может пересчитать progress в базе, если поля `tasks_total`, `tasks_done`, `progress_percent` хранятся в `Board`.

Правила расчёта:

- `tasks_total` = задачи в active columns `kind=board`;
- `tasks_done` = задачи в active column `Done` с `kind=board`;
- задачи в Backlog не входят в progress;
- задачи в Archive column не входят в progress;
- задачи в archived columns не входят в progress.

`board_progress` не возвращать в move response.

   - изменить чужую задачу на своей доске;
   - назначить исполнителя;
   - изменить checklist;
   - переместить задачу.
2. Проверить, что операции разрешены.
3. Под пользователем без доступа к доске попробовать те же действия.
4. Проверить `403`.
