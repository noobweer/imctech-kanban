# 3. Изменение модели Column для поддержки Backlog

В актуальной модели `Column` уже есть обычные Kanban-колонки доски: `To Do`, `In Progress`, `Done` и другие пользовательские колонки.

По новому заданию Backlog должен быть отдельной колонкой в таблице `Column`, но на frontend он отображается как отдельная вкладка вне основной доски.

Чтобы backend мог надёжно отличать обычные колонки доски от специальной backlog-колонки, нужно расширить модель `Column`.

Добавить в `Column` поле:

- `kind` — тип колонки.

Значения `kind`:

- `board` — обычная колонка Kanban-доски;
- `backlog` — специальная колонка Backlog.

Реализовать `kind` через Django choices / enum, а не как свободную строку.

Пример:

```python
class ColumnKind(models.TextChoices):
    BOARD = "board", "Board"
    BACKLOG = "backlog", "Backlog"
```

Поле:

```python
kind = models.CharField(
    max_length=20,
    choices=ColumnKind.choices,
    default=ColumnKind.BOARD,
)
```

Требования:

- `kind` — новое поле, которого не было в исходной модели;
- оно добавляется именно из-за нового требования: Backlog хранится как `Column`, но логически отличается от обычных Kanban-колонок;
- для всех уже существующих колонок выставить `kind = "board"` через default/migration;
- новые обычные колонки должны создаваться с `kind = "board"`;
- backlog-колонка должна создаваться с `kind = "backlog"`;
- у одной доски должна быть максимум одна active backlog-колонка;
- обычные endpoint’ы колонок доски не должны ломаться после добавления `kind`;
- если frontend запрашивает обычные колонки доски, по умолчанию не нужно смешивать их с backlog-колонкой;
- если нужен список backlog-задач, использовать backlog-колонку отдельно;
- если в текущей кодовой базе уже есть другой способ отличать backlog-колонку, использовать существующий способ и не дублировать `kind`.

# 4. Где нужно учесть Column.kind

После добавления `kind` нужно обновить все места, где это влияет на систему.

## 4.1. Миграции

Нужна миграция, которая:

- добавляет `Column.kind`;
- всем существующим колонкам выставляет `kind = "board"`.

## 4.2. Django Admin для Column

В admin для `Column` добавить `kind`.

В `list_display` добавить:

- `kind`

В `list_filter` добавить:

- `kind`

Это нужно, чтобы в админке было видно, какая колонка обычная, а какая backlog.

## 4.3. Column response schemas

В API-ответах по колонке добавить:

- `kind`

Например:

```json
{
  "id": "uuid",
  "board_id": "uuid",
  "board_name": "Test Board",
  "name": "Backlog",
  "kind": "backlog",
  "position": 1,
  "status": "active",
  "sum_tasks": 0,
  "created_at": "2026-05-17T10:00:00Z",
  "updated_at": "2026-05-17T10:00:00Z"
}
```

## 4.4. Column create/update schemas

Обычный endpoint создания колонок:

`POST /api/boards/{board_id}/columns`

должен по умолчанию создавать:

```text
kind = board
```

Не давать frontend свободно создавать `kind=backlog` через обычный endpoint создания колонок, если для этого нет отдельной безопасной логики.

Через `PATCH /api/columns/{column_id}` не разрешать менять `kind`, если нет явной необходимости. Изменение `kind` может сломать логику доски.

## 4.5. Endpoint списка колонок

`GET /api/boards/{board_id}/columns`

По умолчанию должен возвращать обычные колонки доски:

```text
kind = board
```

Backlog не должен внезапно отображаться рядом с `To Do`, `In Progress`, `Done` в обычном board view.

Желательно поддержать фильтр:

```text
GET /api/boards/{board_id}/columns?kind=board
GET /api/boards/{board_id}/columns?kind=backlog
GET /api/boards/{board_id}/columns?kind=all
```

Если `kind=all` не реализуется — указать это в отчёте.

## 4.6. Дефолтные колонки

Endpoint дефолтных колонок:

`POST /api/boards/{board_id}/columns/defaults`

должен создавать:

- `To Do` с `kind=board`;
- `In Progress` с `kind=board`;
- `Done` с `kind=board`.

Он НЕ должен создавать Backlog.

Backlog создаётся логикой Task/backlog, когда создаётся задача без `column_id` или когда отдельно требуется backlog.

## 4.7. Move column

`POST /api/columns/{column_id}/move`

Пересчитывать позиции только внутри одного `kind`.

Правила:

- двигаем `kind=board` колонку — пересчитываем только `kind=board` колонки этой доски;
- backlog-колонку через обычный move endpoint лучше запретить для MVP;
- если move backlog всё же поддерживается, он не должен затрагивать позиции обычных board-колонок.

## 4.8. Position uniqueness

После появления `kind` уникальность позиции должна учитывать тип колонки.

Правило:

- у одной доски не должно быть двух active колонок с одинаковым `position` внутри одного `kind`.

То есть логически:

```text
board + kind + position + active
```

Backlog с `position=1` не должен конфликтовать с `To Do` с `position=1`, потому что они относятся к разным `kind`.

## 4.9. Backlog uniqueness

У одной доски должна быть максимум одна active колонка:

```text
kind = backlog
status = active
```

Реализовать защиту:

- при создании backlog проверять, есть ли уже active backlog этой доски;
- если есть — использовать существующую backlog-колонку;
- не создавать вторую active backlog-колонку.

Если возможно без сильного усложнения, добавить DB-level constraint.  
Если сложно из-за SQLite/текущей архитектуры, сделать проверку на уровне service/endpoint logic и указать это в отчёте.

# 5. Создание Backlog-колонки

При создании задачи через доску без `column_id`:

- найти active backlog-колонку этой доски;
- если backlog-колонки нет — создать её автоматически:
  - `name = "Backlog"`;
  - `kind = "backlog"`;
  - `status = "active"`;
  - `position = 1` внутри backlog-kind;
- создать задачу в этой backlog-колонке.

Важно:

- не создавать несколько active backlog-колонок для одной доски;
- не смешивать backlog с обычными board-колонками;
- не ломать существующие To Do / In Progress / Done.

# 6. Модель Task

Добавить модель `Task`.

Поля:

- `id` — UUID, primary key;
- `column` — ForeignKey на `Column`;
- `title` — string;
- `content` — text, Markdown-строка, может быть пустой;
- `priority` — integer, default `0`, диапазон `0..3`;
- `deadline` — datetime, nullable/optional;
- `status` — enum/string:
  - `active`
  - `archived`
- `assignees` — ManyToMany к пользователям;
- `owner` — ForeignKey на пользователя, создатель задачи;
- `position` — integer, позиция задачи внутри колонки;
- `tags` — JSONField, список строк, default `[]`;
- `checklist` — JSONField, список пунктов чеклиста, default `[]`;
- `created_at`;
- `updated_at`.

Важно:

- стадия работы задачи определяется колонкой;
- `status` отвечает только за active/archived;
- Markdown хранится в `content` как обычная строка;
- отдельную таблицу `ChecklistItem` в MVP не создавать;
- отдельную таблицу `Tag` в MVP не создавать;
- полноценное drag-and-drop перемещение задач между колонками относится к следующей задаче спринта 2;
- поле `links` не добавлять, если его нет в актуальной документации сущностей.

# 7. Markdown content

Поле `content` хранит Markdown как обычный текст.

Backend НЕ должен парсить Markdown.

Frontend может использовать Markdown-форматирование:

- Bold;
- Headings 1/2/3;
- Default Text;
- Bullet List.

Backend должен:

- принять строку;
- сохранить строку;
- вернуть строку.

# 8. Checklist

Поле `checklist` хранится как JSON-list внутри `Task`.

Структура одного пункта:

```json
{
  "title": "Сделать модель Task",
  "is_done": false
}
```

Правила:

- отдельную таблицу `ChecklistItem` в MVP не создавать;
- `checklist` по умолчанию `[]`;
- порядок пунктов хранится порядком элементов в JSON-массиве;
- каждый пункт должен иметь:
  - `title`;
  - `is_done`;
- `title` — непустая строка;
- `is_done` — boolean.

Для карточек задач в response добавить вычисляемые поля:

- `checklist_done_count`;
- `checklist_total_count`.

Пример:

```json
{
  "checklist": [
    {
      "title": "Create model",
      "is_done": true
    },
    {
      "title": "Add validation",
      "is_done": false
    }
  ],
  "checklist_done_count": 1,
  "checklist_total_count": 2
}
```

В базу `checklist_done_count` и `checklist_total_count` сохранять не нужно. Считать из JSON при ответе.

# 10. Assignees

`assignees` — ManyToMany к пользователям.

Правила:

- поле опциональное;
- при создании/обновлении принимать список username;
- все username должны существовать;
- все assignees должны иметь доступ к доске:
  - быть owner доски;
  - или быть в `Board.members`;
  - или быть staff/admin;
- нельзя назначить пользователя, который не состоит в команде доски;
- если `assignees` не передан — список пустой.

Важно:

- отдельные endpoint’ы AssignUser / UnassignUser можно не делать в этой задаче;
- полноценное назначение исполнителей в жизненном цикле задачи относится к задаче 2 спринта 2;
- в этой задаче достаточно поддержать assignees в create/update Task.

# 12. Правила endpoint’ов

## `GET /api/boards/{board_id}/tasks`

Возвращает задачи всех колонок доски.

Доступ:

- owner;
- member;
- staff/admin.

Фильтры:

- `status=active|archived`;
- `column_id`;
- `column_kind=backlog|board`;
- `priority=0|1|2|3`;
- `assignee=username`;
- `tag=backend`;
- `search=text`.

Поиск `search` минимум по:

- `title`;
- `content`.

Если часть фильтров не реализована — указать это в отчёте.

## `GET /api/boards/{board_id}/backlog/tasks`

Возвращает задачи backlog-колонки доски.

Доступ:

- owner;
- member;
- staff/admin.

Фильтры:

- `status`;
- `priority`;
- `assignee`;
- `tag`;
- `search`.

Сортировка:

- `position ASC`;
- затем `created_at ASC`, если нужно.

## `GET /api/columns/{column_id}/tasks`

Возвращает задачи конкретной колонки.

Требования:

- пользователь должен иметь доступ к доске этой колонки;
- возвращать только задачи этой колонки;
- сортировать по `position ASC`;
- поддержать фильтр `status`.

## `GET /api/tasks/{task_id}`

Возвращает одну задачу.

Пользователь должен иметь доступ к доске, к которой относится задача.

Ответ должен включать:

- `id`
- `board_id`
- `board_name`
- `column_id`
- `column_name`
- `column_kind`
- `title`
- `content`
- `priority`
- `deadline`
- `status`
- `assignees`
- `owner_username`
- `position`
- `tags`
- `checklist`
- `checklist_done_count`
- `checklist_total_count`
- `created_at`
- `updated_at`

## `POST /api/boards/{board_id}/tasks`

Создаёт задачу на доске.

Если `column_id` не передан — задача создаётся в Backlog.

Минимальное тело:

```json
{
  "title": "Prepare task API"
}
```

Расширенное тело:

```json
{
  "title": "Implement Task API",
  "content": "## Description\nCreate CRUD endpoints for Task.",
  "column_id": "uuid-column-id",
  "priority": 2,
  "deadline": "2026-05-17T20:00:00Z",
  "tags": ["backend", "api"],
  "checklist": [
    {
      "title": "Create model",
      "is_done": false
    },
    {
      "title": "Add migrations",
      "is_done": false
    }
  ],
  "assignees": ["test_student"]
}
```

Правила:

- создавать задачу может пользователь с доступом к доске;
- текущий пользователь становится `owner`;
- если `column_id` передан — колонка должна принадлежать этой доске;
- если `column_id` не передан — использовать Backlog;
- если Backlog отсутствует — создать Backlog-колонку;
- если `position` не передан — задача добавляется в конец выбранной колонки;
- опциональные поля можно не передавать.

## `POST /api/columns/{column_id}/tasks`

Создаёт задачу сразу в конкретной колонке.

Правила:

- пользователь должен иметь доступ к доске этой колонки;
- текущий пользователь становится `owner`;
- если `position` не передан — задача добавляется в конец колонки.

## `PATCH /api/tasks/{task_id}`

Можно менять:

- `title`;
- `content`;
- `priority`;
- `deadline`;
- `status`;
- `assignees`;
- `tags`;
- `checklist`.

В этой задаче не делать полноценный drag-and-drop move между колонками, если это не требуется уже существующей архитектурой.

Если `column_id` и `position` через PATCH не поддерживаются — указать в отчёте.

## `POST /api/tasks/{task_id}/archive`

Архивирует задачу:

```text
status = archived
```

Физически не удалять.

## `POST /api/tasks/{task_id}/restore`

Восстанавливает задачу:

```text
status = active
```

## `DELETE /api/tasks/{task_id}`

Для MVP реализовать как soft-delete:

```text
status = archived
```

Физически не удалять.

# 13. Права доступа

Использовать простую командную модель доступа:

- staff/admin видит и управляет всеми задачами;
- owner доски видит и управляет всеми задачами своей доски;
- member доски видит все задачи доски;
- member доски может создавать задачи;
- member доски может редактировать любые задачи этой доски;
- member доски может архивировать и восстанавливать любые задачи этой доски;
- пользователь без доступа к доске не может видеть, создавать или менять задачи этой доски.

Роли `student` и `mentor` пока не усложнять.  
Если пользователь состоит в команде доски, он может работать с задачами этой доски.

# 14. Валидация

## Общая

- `title` обязателен;
- `title` не должен быть пустым;
- `content` может быть пустой строкой;
- `priority` должен быть integer `0..3`;
- если `priority` не передан — default `0`;
- `deadline` может быть `null`;
- если `deadline` передан — должен быть валидным datetime;
- `status` только `active` или `archived`;
- `position` — положительное число;
- нельзя создать задачу в колонке чужой доски;
- нельзя получить задачу без доступа к доске;
- нельзя редактировать задачу без доступа к доске.

## Checklist

Если передан `checklist`:

- должен быть list/array;
- каждый элемент должен быть object;
- у каждого элемента должен быть `title`;
- `title` — непустая строка;
- у каждого элемента должен быть `is_done`;
- `is_done` — boolean;
- лишние поля либо игнорировать, либо возвращать ошибку. Выбрать один вариант и указать в отчёте.

Если `checklist` не передан — использовать `[]`.

## Tags

Если передан `tags`:

- должен быть list/array;
- каждый элемент — непустая строка.

Если `tags` не передан — использовать `[]`.

## Assignees

Если передан `assignees`:

- должен быть list/array username;
- каждый username должен существовать;
- каждый assignee должен иметь доступ к доске;
- нельзя назначить пользователя вне команды доски.

# 16. Обновление Board progress

Так как появилась сущность `Task`, нужно обновлять или уметь пересчитывать поля доски:

- `tasks_total`;
- `tasks_done`;
- `progress_percent`.

Минимальная MVP-логика:

- `tasks_total` = количество active задач в `kind=board` колонках доски;
- `tasks_done` = количество active задач в `kind=board` колонке `Done`;
- задачи в `kind=backlog` НЕ входят в progress;
- `progress_percent = tasks_done / tasks_total * 100`, если `tasks_total > 0`, иначе `0`.

Реализация может быть:

- отдельная helper-функция `recalculate_board_progress(board)`;
- вызов после создания/обновления/архивации/восстановления задачи;
- или endpoint/метод для ручного пересчёта, если так проще в текущей архитектуре.

В отчёте указать выбранный вариант.

# 19. Ручная проверка через Swagger

## Предусловия

Есть пользователь:

```json
{
  "username": "test_mentor",
  "password": "12345678",
  "name": "Test Mentor",
  "role": "mentor"
}
```

Есть второй пользователь:

```json
{
  "username": "test_student",
  "password": "12345678",
  "name": "Test Student",
  "role": "student"
}
```

Есть доска:

```json
{
  "name": "Test Board"
}
```

Есть обычные колонки:

```text
To Do
In Progress
Done
```

Backlog создаётся автоматически как `Column.kind = backlog` при создании задачи без `column_id`, если его ещё нет.

Все запросы выполнять с JWT:

```text
Authorization: Bearer <access_token>
```

## Проверить изменение Column.kind

Проверить:

- у старых колонок `kind = board`;
- в admin у Column видно поле `kind`;
- в response колонок есть `kind`;
- `GET /api/boards/{board_id}/columns` по умолчанию не смешивает Backlog с обычными колонками;
- дефолтные колонки создаются с `kind=board`.

## Проверить создание задачи в Backlog

1. Создать задачу без `column_id`:

```json
{
  "title": "Prepare backend task"
}
```

2. Проверить:

- задача создана;
- если Backlog не существовал, он был создан как `Column.kind = backlog`;
- задача попала в Backlog;
- `owner` = текущий пользователь;
- `status = active`;
- `priority = 0`;
- `tags = []`;
- `checklist = []`;
- `assignees = []`.

## Проверить создание задачи в конкретной колонке

1. Создать задачу с `column_id` обычной board-колонки:

```json
{
  "title": "Implement Task API",
  "column_id": "uuid-column-id",
  "priority": 2
}
```

2. Проверить:

- задача создана в указанной колонке;
- колонка принадлежит нужной доске;
- задача отображается в `GET /api/columns/{column_id}/tasks`.

## Проверить расширенные поля

Создать задачу:

```json
{
  "title": "Implement task checklist",
  "content": "## Goal\nAdd checklist JSON validation.",
  "priority": 3,
  "deadline": "2026-05-17T20:00:00Z",
  "tags": ["backend", "task"],
  "checklist": [
    {
      "title": "Create model",
      "is_done": true
    },
    {
      "title": "Add validation",
      "is_done": false
    }
  ],
  "assignees": ["test_student"]
}
```

Проверить:

- `content` сохраняется как Markdown-строка;
- `tags` сохраняются как JSON-list;
- `checklist` сохраняется как JSON-list;
- `checklist_done_count = 1`;
- `checklist_total_count = 2`;
- assignees назначаются только если пользователь имеет доступ к доске.

## Проверить права member

1. Добавить `test_student` в доску.
2. Под JWT `test_student`:
   - создать задачу;
   - отредактировать свою задачу;
   - отредактировать чужую задачу на этой же доске;
   - архивировать задачу на этой же доске.

Ожидаемо:

- member может работать с задачами своей доски;
- пользователь без доступа к доске не может видеть или менять задачи.

## Проверить search

Вызвать:

```text
GET /api/boards/{board_id}/tasks?search=backend
GET /api/boards/{board_id}/backlog/tasks?search=backend
```

Проверить:

- поиск работает по `title`;
- поиск работает по `content`.

## Проверить filters

Проверить:

```text
GET /api/boards/{board_id}/tasks?status=active
GET /api/boards/{board_id}/tasks?status=archived
GET /api/boards/{board_id}/tasks?priority=3
GET /api/boards/{board_id}/tasks?tag=backend
GET /api/boards/{board_id}/tasks?assignee=test_student
GET /api/boards/{board_id}/tasks?column_kind=backlog
GET /api/boards/{board_id}/tasks?column_kind=board
```

## Проверить archive/restore/delete

Проверить:

```text
POST /api/tasks/{task_id}/archive
POST /api/tasks/{task_id}/restore
DELETE /api/tasks/{task_id}
```

Ожидаемо:

- archive ставит `status = archived`;
- restore ставит `status = active`;
- DELETE физически не удаляет, а архивирует.

## Проверить невалидный checklist

Эти запросы должны вернуть ошибку:

```json
{
  "title": "Bad checklist",
  "checklist": "not a list"
}
```

```json
{
  "title": "Bad checklist",
  "checklist": [
    {
      "title": "",
      "is_done": false
    }
  ]
}
```

```json
{
  "title": "Bad checklist",
  "checklist": [
    {
      "title": "Check item",
      "is_done": "yes"
    }
  ]
}
```
